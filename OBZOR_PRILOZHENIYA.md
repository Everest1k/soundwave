# Обзор приложения «Soundwave» (музыкальный плеер)

Документ-разбор всего приложения для курсовой работы: архитектура, функционал,
работа плеера, загрузка/выгрузка треков, плейлисты, избранное, авторизация.

---

## 1. Что это за приложение

**Soundwave** — Android-приложение, музыкальный стриминговый/локальный плеер.
Пользователь может:

- зарегистрироваться / войти (есть роль обычного пользователя и роль администратора);
- слушать треки, которые подгружаются из общей папки на **Яндекс.Диске** (онлайн)
  и/или импортируются из памяти телефона (локально);
- управлять воспроизведением (плеер с очередью, перемотка, следующий/предыдущий,
  мини-плеер внизу экрана);
- искать треки и видеть недавно прослушанные;
- добавлять треки в **«Любимые»** (избранное у каждого пользователя своё);
- создавать **плейлисты** и наполнять их треками;
- администратор дополнительно может переименовывать и удалять треки.

### Технологический стек

| Категория | Технология |
|---|---|
| Язык | **Java** (не Kotlin) |
| Платформа | Android, `minSdk 24`, `targetSdk 36` |
| UI | XML-разметки + View Binding, фрагменты, `BottomNavigationView` |
| База данных | **Room** (SQLite-обёртка), версия схемы 4 |
| Асинхронность | `ExecutorService` (один поток) + `Handler` главного потока |
| Реактивность | **LiveData** (наблюдение за данными из БД и состоянием плеера) |
| Загрузка картинок | **Glide** |
| Сеть | «голый» `HttpURLConnection` + парсинг JSON через `org.json` |
| Плеер | системный `android.media.MediaPlayer` |
| Хранение настроек/сессии | `SharedPreferences` |

Архитектура — упрощённый вариант **MVVM**: репозитории + LiveData как источник
данных, фрагменты/активити как View, ViewModel-слой почти не используется
(его роль частично берут на себя репозитории и `PlayerManager`).

## 3. Точка входа и навигация между экранами

### Поток запуска

1. **`SoundwaveApp.onCreate()`** — при старте процесса заранее создаются
   синглтоны `TrackRepository` и `PlayerManager` (чтобы база и плеер были готовы).
2. Запускается **`LoginActivity`** (она помечена `LAUNCHER` в манифесте).
   - Если пользователь уже залогинен (`AuthManager.isLoggedIn()`), сразу
     переходит в `MainActivity`.
3. **`MainActivity`** — главный экран. Если сессии нет, отбрасывает обратно на логин.

### Нижняя навигация (`MainActivity`)

Три вкладки через `BottomNavigationView`:

- **Главная** → `HomeFragment`
- **Поиск** → `SearchFragment`
- **Библиотека** → `LibraryFragment`

Фрагменты не пересоздаются при переключении: используется паттерн
`hide()/show()` (метод `showFragment`), поэтому состояние вкладок сохраняется.

Поверх контента всегда висит **мини-плеер** (`view_miniplayer.xml`), общий для
всех вкладок. Внутренние экраны (избранное, плейлисты, детали плейлиста,
добавление треков) открываются как фрагменты поверх с добавлением в back stack.

## 4. Авторизация и роли (`AuthManager`)

Очень простая «учебная» авторизация **полностью на устройстве**, без сервера.

- Данные хранятся в `SharedPreferences` (файл `auth_prefs`):
  - `users_json` — JSON-объект вида `{"логин": "пароль", ...}` (пароли в открытом виде — это упрощение, в реальном приложении так делать нельзя);
  - `current_user_id` — кто сейчас вошёл.
- **Регистрация** (`register`): проверяет, что логин не занят и не пустой,
  нельзя зарегистрировать `admin`. Сразу логинит нового пользователя.
- **Вход** (`login`): возвращает `null` при успехе или строку с текстом ошибки.
- **Администратор** — «зашит» в коде:
  - логин `admin`, пароль `admin1234` (константы в `AuthManager`);
  - его `userId` = `__admin__`.
- **Роли**: `isAdmin()` / `role()` (`"admin"` или `"user"`).
- **Выход** (`logout`): удаляет `current_user_id`.

Важная деталь: **избранное и плейлисты привязаны к `userId`** — у каждого
пользователя свой набор (см. ниже). Сами треки — общие для всех.

## 5. Слой данных (Room)

База **`soundwave.db`**, версия 4, 4 таблицы. Используется
`fallbackToDestructiveMigration()` — при смене версии база просто пересоздаётся
(данные теряются; для учебного проекта приемлемо).

### Таблица `tracks` — трек (`Track`)

| Поле | Тип | Назначение |
|---|---|---|
| `id` | long (PK, autoincrement) | идентификатор |
| `title` | String | название |
| `artist` | String | исполнитель |
| `filePath` | String | источник аудио: `file://...` (локальный) или `https://...` (Яндекс.Диск) |
| `coverPath` | String? | обложка: путь к файлу или URL |
| `durationMs` | long | длительность (для облачных = 0, узнаётся при воспроизведении) |
| `addedAt` | long | время добавления (для сортировки) |
| `isLiked` | boolean | устаревшее поле «лайка» (фактически лайки теперь в таблице `favorites`) |
| `remoteId` | String? | идентификатор файла на Яндекс.Диске (для дедупликации при синхронизации) |

### Таблица `playlists` — плейлист (`Playlist`)

`id`, `name`, `createdAt`, **`ownerId`** (логин владельца — у каждого свои плейлисты).

### Таблица `playlist_tracks` — связь «плейлист ↔ трек» (`PlaylistTrack`)

Составной первичный ключ `{playlistId, trackId}` — классическая
many-to-many связь. Поле `addedAt` задаёт порядок треков в плейлисте.

### Таблица `favorites` — избранное (`Favorite`)

Составной ключ `{userId, trackId}` + `addedAt`. То есть лайк — это запись
«пользователь X любит трек Y». Благодаря этому избранное персональное.

### DAO (запросы)

- **`TrackDao`**: добавить/удалить трек, наблюдать все (`observeAll`), недавние
  (`observeRecent` с лимитом), поиск по названию/исполнителю (`search` через
  `LIKE`), получить по id, найти по `remoteId` (дедуп), переименовать.
- **`PlaylistDao`**: список плейлистов владельца **с числом треков** (подзапрос
  `COUNT`), добавить/убрать трек в плейлист, наблюдать треки плейлиста (через
  `INNER JOIN`), удалить трек из всех плейлистов (при удалении самого трека).
- **`FavoriteDao`**: добавить/убрать лайк, проверить/наблюдать лайк, список
  любимых пользователя (`JOIN tracks ↔ favorites`), счётчик, удалить лайки трека.

### Репозитории (фасады)

- **`TrackRepository`** (синглтон): все операции с треками и избранным.
  - Поиск реализован **двумя способами**: есть DAO-метод `search` с `LIKE`, но
    в репозитории `search()` фильтрует список в памяти через `Transformations.map`
    (регистронезависимо).
  - `observeByIds` — получить треки по списку id **с сохранением порядка**
    (нужно для «недавних»: они должны идти в порядке прослушивания).
  - Все записи в БД (insert/delete/setLiked/rename) — на фоновом потоке `io`.
  - `delete(track)` каскадно чистит связи: убирает трек из всех плейлистов и
    из избранного, потом удаляет сам трек.
- **`PlaylistRepository`** (синглтон): создание/удаление плейлистов,
  добавление/удаление треков, наблюдение списка и содержимого. Все плейлисты
  фильтруются по `ownerId` текущего пользователя.

### «Недавние» (`RecentTracksPrefs`)

Хранятся **не в БД, а в `SharedPreferences`** как строка id через запятую
(максимум 20). При прослушивании трека его id ставится в начало списка
(дубли убираются). Это «история поиска/прослушиваний».

## 6. Плеер (`PlayerManager`)

Сердце приложения. Это **синглтон**, живущий всё время работы процесса
(поэтому музыка продолжает играть при переходах между экранами).

Внутри — системный `MediaPlayer`. Состояние публикуется через **LiveData**,
на которые подписаны и `MainActivity` (мини-плеер), и `PlayerActivity`
(полный экран):

- `currentTrack()` — текущий трек;
- `isPlaying()` — играет/пауза;
- `progressMs()` — текущая позиция (обновляется тикером каждые 500 мс);
- `durationMs()` — длительность трека.

### Очередь воспроизведения

- `setQueue(list, startIndex)` — задать список треков и начать с нужного.
  Везде в UI при клике на трек передаётся **весь видимый список** — так очередь
  становится контекстной (кликнул в плейлисте → очередь = плейлист, кликнул в
  избранном → очередь = избранное и т.д.).
- `playSingle(track)` — очередь из одного трека.
- `next()` — следующий по кругу (`% size`).
- `previous()` — «умная» кнопка: если играем дольше 3 секунд — перематывает в
  начало текущего трека, иначе переходит к предыдущему.
- по завершении трека (`OnCompletionListener`) автоматически вызывается `next()`.

### Управление

- `toggle()` — пауза/продолжить (или старт, если плеер был остановлен);
- `seekTo(ms)` — перемотка (используется ползунком в `PlayerActivity`);
- `stop()` — освобождает `MediaPlayer` (вызывается при выходе из аккаунта).

### Как проигрывается трек (`play`)

1. `MediaPlayer` создаётся/сбрасывается, задаются `AudioAttributes` (тип «музыка»).
2. `setDataSource(uri)` — **одинаково работает и для локального файла, и для
   URL Яндекс.Диска** (MediaPlayer сам умеет стримить по http).
3. `prepareAsync()` — асинхронная подготовка; по готовности
   (`OnPreparedListener`) узнаётся длительность и начинается воспроизведение.
4. Ошибки гасятся `OnErrorListener` (просто ставится пауза).

> Замечание для доработки: реальной фоновой службы (Foreground Service) нет,
> хотя разрешения `FOREGROUND_SERVICE*` в манифесте объявлены. Сейчас плеер
> живёт в памяти процесса, без уведомления-управления. Это очевидная точка
> для улучшения в курсовой.

### Экран плеера (`PlayerActivity`)

Полноэкранный плеер: обложка, название, исполнитель, ползунок (`SeekBar`) с
таймингами (текущее время и оставшееся со знаком «-»), кнопки
play/pause/prev/next, кнопка «лайк» и «назад».

- Подписывается на LiveData плеера.
- Лайк наблюдается через `TrackRepository.observeLiked(trackId)` и
  переподписывается при смене трека; цвет сердечка меняется по состоянию.
- Если текущего трека нет (`null`) — экран закрывается.

### Мини-плеер (в `MainActivity`)

Маленькая панель снизу: обложка, название, исполнитель, кнопка play/pause,
полоса прогресса. Скрывается, когда трека нет. Клик по панели открывает
полноэкранный `PlayerActivity`.

## 7. Загрузка треков «в приложение» (выгрузка контента)

Есть **два независимых источника** треков.

### 7.1. Облачная загрузка с Яндекс.Диска (`YandexDiskLoader`) — основной путь

Это и есть «выгрузка/загрузка треков» в смысле наполнения каталога.

- Ссылка на публичную папку задаётся в **`RemoteTracksConfig.YANDEX_DISK_PUBLIC_LINK`**
  (сейчас: `https://disk.yandex.com/d/97VBe8dYd2q_5A`).
- Используется **публичный API Яндекс.Диска** (без токена/авторизации):
  `https://cloud-api.yandex.net/v1/disk/public/resources?public_key=<ссылка>&limit=500&preview_size=L`.

**Алгоритм синхронизации (`sync`)** — на фоновом потоке:

1. HTTP GET к API, парсинг JSON-ответа (`HttpURLConnection` + `org.json`).
2. Из ответа берётся список файлов (`_embedded.items`).
3. **Первый проход** — собираются картинки (`.jpg/.png/.webp`) в карту
   «имя без расширения → URL обложки». Так обложка сопоставляется треку по имени.
4. **Второй проход** — берутся аудиофайлы (по `media_type=audio` или
   расширению `.mp3/.m4a/.wav/.ogg/.flac`):
   - `remoteId` = путь файла на диске → **дедупликация** через
     `trackDao.findByRemoteId` (повторная синхронизация не плодит дубли);
   - обложка = либо одноимённая картинка из папки, либо `preview` от API;
   - **имя парсится**: если в названии есть `" - "`, то слева — исполнитель,
     справа — название (`Artist - Title`);
   - создаётся `Track` с `filePath = прямой URL файла` и сохраняется в БД.
5. Результат (сколько добавлено / текст ошибки) возвращается в UI-поток через
   `Handler` и колбэк `Callback`.

**Где запускается синхронизация:**

- Автоматически при старте `MainActivity` (если ссылка задана), молча.
- Вручную — кнопкой **«Обновить треки»** на «Главной» (`HomeFragment`),
  с тостами о прогрессе и результате.

> То есть «загрузка треков» = подтягивание метаданных (название, исполнитель,
> обложка, прямая ссылка) в локальную БД. Само аудио **не скачивается на
> устройство** — оно стримится по URL при воспроизведении.

### 7.2. Импорт локального файла (`TrackImporter`)

Утилита для импорта аудио из памяти телефона (через `content://`-Uri):

1. Файл **копируется** во внутреннее хранилище приложения (`filesDir/audio/`),
   имя нормализуется.
2. Через `MediaMetadataRetriever` читаются метаданные: название, исполнитель,
   длительность и **встроенная обложка** (если есть, сохраняется в `filesDir/covers/`).
3. Возвращается готовый `Track` с `filePath = file://...`.

> Замечание: класс `TrackImporter` реализован, но в текущем UI кнопки выбора
> файла, которая бы его вызывала, нет (загрузка идёт через Яндекс.Диск).
> Это «задел» — ещё одна точка для доработки в курсовой (добавить экран
> загрузки своего трека через `SAF`/`ACTION_OPEN_DOCUMENT`).

### Загрузка обложек (`CoverLoader`)

Единая точка: если `coverPath` пустой — плейсхолдер; если начинается с `http` —
грузится по сети через Glide; иначе — как локальный файл. Везде есть
плейсхолдер `cover_placeholder`.

## 8. Экраны и их функционал

### Главная (`HomeFragment`)

- Сетка карточек треков (`StaggeredGridLayoutManager`, горизонтальная,
  адаптер `TrackCardAdapter`).
- Показывает **20 недавно добавленных** треков (`observeRecent(20)`) как
  «рекомендации». Пусто → подсказка.
- Кнопка **«Обновить треки»** → синхронизация с Яндекс.Диском.
- Клик по карточке → ставит весь список как очередь и играет с этой позиции.

### Поиск (`SearchFragment`)

- Поле ввода с `TextWatcher` — поиск «на лету».
- Пустой запрос → показываются **недавно прослушанные** (из `RecentTracksPrefs`
  через `observeByIds`), с заголовком и кнопкой «очистить».
- Непустой запрос → `TrackRepository.search()` (фильтр по названию/исполнителю).
- Адаптер `TrackRowAdapter` (треки строками). При клике трек добавляется в
  «недавние» и запускается воспроизведение.

### Библиотека (`LibraryFragment`)

- Два раздела-кнопки: **«Мои любимые»** (с числом треков) и **«Плейлисты»**
  (с числом). Счётчики наблюдаются из БД.
- Показывает роль и имя текущего пользователя.
- Кнопка **«Выйти»**: останавливает плеер, делает `logout`, возвращает на
  логин с очисткой стека активити.
- Числительные склоняются по-русски (`1 трек / 2 трека / 5 треков`).

### Любимые (`FavoritesFragment`)

- Список любимых треков текущего пользователя (`observeLiked` →
  `JOIN favorites ↔ tracks`). Пусто → подсказка. Клик → играть очередью.

### Плейлисты (`PlaylistsFragment`)

- Список плейлистов пользователя (`PlaylistsAdapter`, с числом треков).
- Кнопка **«Новый плейлист»** → диалог с вводом имени → создание в БД.
- Клик по плейлисту → экран деталей.

### Детали плейлиста (`PlaylistDetailFragment`)

- Заголовок = имя плейлиста, список треков (`observeTracks`).
- Кнопка **«Добавить треки»** → экран добавления.
- Кнопка **«Удалить»** → подтверждение → удаление плейлиста.
- Клик по треку → играть очередью (очередь = этот плейлист).

### Добавление треков в плейлист (`AddTracksFragment`)

- Показывает **все** треки каталога. Клик по треку → добавляет его в плейлист
  (`addTrack`) и показывает тост «добавлено».

## 9. Функции администратора

Админ определяется через `AuthManager.isAdmin()`. В адаптерах треков
(`TrackRowAdapter`, `TrackCardAdapter`) на **долгое нажатие** по треку, если
текущий пользователь — админ, открывается `AdminTrackActions` —
контекстное меню (`PopupMenu`):

- **Переименовать** — диалог с новым названием (`TrackRepository.rename`);
- **Удалить** — подтверждение → каскадное удаление трека из БД, плейлистов и
  избранного.

Для обычного пользователя долгое нажатие ничего не делает.

## 10. Потоки данных (как всё связано)

```
                 Яндекс.Диск (публичная папка)
                          │  HTTP GET (JSON)
                          ▼
                 YandexDiskLoader.sync()  ──► пишет Track в БД
                                                   │
   TrackImporter (локальный файл) ──► Track ──►    │
                                                   ▼
                          ┌──────────  Room (soundwave.db)  ──────────┐
                          │  tracks · playlists · playlist_tracks ·   │
                          │             favorites                     │
                          └───────────────────┬───────────────────────┘
                                              │ LiveData (наблюдение)
            ┌──────────────┬──────────────────┼───────────────┬──────────────┐
            ▼              ▼                  ▼               ▼              ▼
        HomeFragment  SearchFragment   FavoritesFragment  Playlists...   PlayerActivity
            │ клик по треку → setQueue(список, позиция)                       ▲
            └───────────────────────────► PlayerManager (MediaPlayer) ───────┘
                                              │ LiveData: track/playing/progress
                                              ▼
                                     мини-плеер в MainActivity
```

Ключевые идеи архитектуры:

1. **Единый источник правды — Room**, UI только наблюдает LiveData и
   автоматически обновляется при изменении данных.
2. **Плеер — глобальный синглтон**, общий для всех экранов, со своим набором
   LiveData для состояния.
3. **Очередь воспроизведения контекстная** — определяется тем списком, из
   которого пользователь запустил трек.
4. **Персонализация по `userId`/`ownerId`** — избранное и плейлисты у каждого
   пользователя свои, каталог треков — общий.

## 11. Разрешения и манифест

- `INTERNET`, `ACCESS_NETWORK_STATE` — для загрузки с Яндекс.Диска и стриминга.
- `READ_MEDIA_AUDIO` / `READ_EXTERNAL_STORAGE` (до Android 12) — для импорта
  локального аудио.
- `FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_MEDIA_PLAYBACK` — объявлены, но
  фоновая служба пока не реализована.
- Активити: `LoginActivity` (launcher), `MainActivity`, `PlayerActivity`.

## 12. Сильные и слабые стороны (что можно улучшить в курсовой)

**Сильные стороны**

- Чистое разделение слоёв (data / player / remote / ui).
- Реактивный UI на LiveData — мало ручного обновления списков.
- Грамотная связь many-to-many для плейлистов и персональное избранное.
- Дедупликация треков при повторной синхронизации.

**Что стоит доработать / переделать**

- **Безопасность авторизации**: пароли в открытом виде в `SharedPreferences`,
  «зашитый» админ. → хеширование, нормальный бэкенд или хотя бы `EncryptedSharedPreferences`.
- **Фоновое воспроизведение**: нет `MediaSessionService`/Foreground Service и
  уведомления-плеера — музыка не управляется из шторки и может убиваться системой.
- **Современный плеер**: `MediaPlayer` устарел — переход на **ExoPlayer/Media3**
  даст кэширование, гэплесс, лучшее стриминг-поведение.
- **Сеть**: «голый» `HttpURLConnection` → Retrofit/OkHttp + Gson/Moshi.
- **Поиск**: дублирование логики (есть и SQL-`LIKE`, и фильтр в памяти) — оставить один.
- **Поле `Track.isLiked`** фактически не используется (лайки в таблице
  `favorites`) — мёртвый код, можно убрать.
- **Импорт локальных файлов** реализован (`TrackImporter`), но не подключён к UI.
- **Миграции БД**: сейчас `fallbackToDestructiveMigration` — данные теряются
  при обновлении схемы; для реального приложения нужны настоящие миграции.
- **ViewModel-слой** почти отсутствует — можно ввести полноценные ViewModel.
